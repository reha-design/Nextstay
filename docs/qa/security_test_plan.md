# 🛡️ Nextstay 보안 및 품질 보증(QA) 분석 보고서

**수석 분석 에이전트:** Antigravity  
**분석 대상:** 회원가입(Signup) 및 로그인(Login) 프로세스  
**분석 목적:** SQL 인젝션 및 비정상 문자열 입력에 대한 계층별 독립 방어 로직 검증

---

## 1. 공격 페이로드(Payload) 데이터 셋

| 카테고리 | 테스트 페이로드 (예시) | 공격 목표 및 의도 |
| :--- | :--- | :--- |
| **SQL Injection** | `' OR 1=1 --`, `admin' #`, `'; DROP TABLE member; --` | DB 쿼리 구조 조작을 통한 인증 우회 및 데이터 파괴 |
| **악성 스크립트 (XSS)** | `<script>alert(1)</script>`, `<img src=x onerror=alert(1)>` | 사용자 브라우저상에서 악성 자바스크립트 실행 |
| **특수 기호 & 유효성** | `name%20@!`, `admin&email=evil@com`, `\ \0 \n \r \t` | 서버 로직 오작동 유도 및 시스템 경로/메모리 오염 시도 |
| **비정상 형식/길이** | `A` * 5000 (Over 4KB), `null`, `[]`, `{}` (타입 불일치) | Buffer Overflow 유도 및 파싱 에러(500) 유발을 통한 취약점 탐색 |

## 2. 보안 검증 및 방어 동작 테스트 계획

| 검증 계층 | 점검 대상 필드 | 테스트 필드/페이로드 | 기대되는 정상 방어 동작 | 취약 시 발생 위험 |
| :--- | :--- | :--- | :--- | :--- |
| **Front** | 이메일/비밀번호 | `admin' #` (Copy & Paste) | 제출 시 `Vuelidate` 또는 `Regex`에 의해 전송 차단 후 경고 노출 | 악성 페이로드가 서버로 전달되어 불필요한 부하 발생 |
| **Front** | 전화번호 (Input) | `abc-defg-hijk` (직접 타이핑) | `v-phone` 커스텀 디렉티브에 의해 숫자 외 키 입력 무시 | 데이터 형식 불일치로 인한 서버 측 런타임 에러 |
| **Back** | `/api/v1/auth/signup` | JSON Body: `{ "name": "'; DROP..." }` | **400 Bad Request** 반환 및 에러 메시지에 DB 구조 노출 금지 | SQL 파싱 오류로 인한 DB 테이블 삭제 및 데이터 유출 |
| **Back** | `/api/v1/auth/login` | Email: `' OR 1=1 --` | **401 Unauthorized** 혹은 **400 Bad Request** 처리 | 관리자 계정 권한 탈취 및 인증 메커니즘 무력화 |
| **Back** | 공통 API | 매우 긴 문자열 (5000자 이상) | 서버 단에서 'Max Length' 위반으로 처리 (**400**) | Memory Leak 또는 DoS(Denial of Service) 공격 노출 |

## 3. 핵심 점검 시나리오

### **[시나리오 A] 프론트엔드 입력 원천 차단 테스트**
- **수행**: `v-phone` 등 입력 제한 로직이 적용된 필드에 영문/특수문자를 직접 타이핑.
- **기대 결과**: 타이핑 시도 시 텍스트 박스에 아무런 글자도 찍히지 않아야 함.

### **[시나리오 B] 강제 우회 입력(Copy & Paste) 테스트**
- **수행**: 메모장에 `' OR 1=1 --`를 적은 뒤 이메일 필드에 강제 붙여넣기 후 '회원가입' 버튼 클릭.
- **기대 결과**: 버튼 클릭 즉시 클라이언트 단 유효성 검사 로직이 작동하여 "이메일 형식이 올바르지 않습니다" 경고 발생.

### **[시나리오 C] API 직접 타격(Direct Attack) 테스트**
- **수행**: Postman을 사용하여 `POST /api/v1/auth/signup`에 비정상적인 데이터(`password` 필드에 특수문자 대량 포함 등)를 JSON으로 전송.
- **기대 결과**: 프론트엔드 통제 없이 들어온 요청에 대해 서버가 `@Valid` 및 커스텀 Validation 로직에 의해 **400 Bad Request**를 응답해야 하며, 스택 트레이스(Stack Trace)가 노출되지 않아야 함.

---

## 4. 분석 결과 제언
- **보안 강화**: 현재 백엔드 엔티티 및 DTO에서 `@Size`, `@Pattern` 등을 적극 활용하여 DB 쿼리 실행 전 메모리 단에서 1차 검증을 수행해야 합니다.
- **로깅**: 필터링된 공격 시도는 `SecurityContext` 로그에 기록하여 반복적인 공격 IP를 조기에 식별할 수 있도록 구성하는 것을 권장합니다.
