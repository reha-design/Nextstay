# 트랜잭션 및 동시성 제어 트러블슈팅 리포트

예약 시스템 개발 중 발생한 **이중 예약(Double Booking)** 문제와 이를 해결하기 위한 기술적 의사결정 과정을 기록합니다.

---

## 1. 문제 현상
- **상황**: 5개의 스레드가 동일한 시간대의 동일한 객실에 대해 동시에 예약 요청을 보냄.
- **기대 결과**: 비관적 락(`PESSIMISTIC_WRITE`)에 의해 단 1건의 예약만 성공하고, 나머지는 409 Conflict로 실패해야 함.
- **실제 결과**: 5건 모두 예약 성공 (데이터 무결성 파괴).

## 2. 원인 분석

### A. MySQL Isolation Level (REPEATABLE_READ) 문제
- MySQL의 기본 격리 수준은 `REPEATABLE_READ`입니다.
- **현상**: 첫 번째 트랜잭션이 락을 잡고 예약을 완료(Commit)한 후 락을 해제하더라도, 대기 중이던 두 번째 트랜잭션은 **자신의 트랜잭션이 시작된 시점의 스냅샷**을 보기 때문에 이미 생성된 예약을 인지하지 못하고 중복 예약을 생성함.

### B. 가시성 및 플러시 타이밍
- `JPA`의 `save()`는 트랜잭션 종료 시점에 쓰기 지연 저장소의 커맨드를 실행합니다. 
- 락이 해제되는 시점과 실제 데이터가 타 트랜잭션에게 보이기 시작하는 시점 사이에 미세한 간극이 존재할 수 있음이 확인되었습니다.

### C. Spring Data JPA의 제약 사항
- `@Query` 파라미터에 Kotlin의 디폴트 인자(`List<BookingStatus> = ...`)를 사용하려 했으나, Spring Data JPA는 이를 지원하지 않아 런타임 빈 생성 오류가 발생했습니다.

---

## 3. 해결 방안 및 적용

### ✅ Isolation Level 하향 조정 (`READ_COMMITTED`)
- 예약 생성 메서드(`createBooking`)에 한해 격리 수준을 `READ_COMMITTED`로 명시적으로 변경했습니다.
- 이를 통해 락을 획득한 직후, **다른 트랜잭션이 커밋한 최신 데이터**를 읽어올 수 있게 되어 중복 예약 검증이 정확하게 이루어집니다.

### ✅ 비관적 락 위치 최적화 및 PK 기반 변경
- 트랜잭션 내에서 다른 엔티티(Guest 등)를 조회하기 전에 **객실(Room)에 대한 락을 가장 먼저 획득**하도록 순서를 조정했습니다.
- 비즈니스 키(`roomNo`) 대신 **PK(`id`) 기반의 락 조회**를 추가하여 MySQL의 인덱스 기반 Row-level Lock이 더 확실하게 작동하도록 보장했습니다.

### ✅ `saveAndFlush()` 사용
- 예약을 저장한 후 트랜잭션 종료를 기다리지 않고 즉시 DB에 반영(`Flush`)하도록 변경하여, 락이 해제되는 시점과 데이터의 물리적 존재 시점을 일치시켰습니다.

---

## 4. 최종 결과 검증
- **테스트 결과**: 5개 동시 요청 시 1건 성공, 4건 충돌 발생 확인 (성공 로그 및 `test_result.log` 반영).
- **부수적 효과**: `ddl-auto: create-drop` 적용을 통해 테스트마다 깨끗한 DB 상태를 유지하여 테스트 재현성을 확보함.

---

> [!IMPORTANT]
> 금융/예약 시스템과 같이 정밀한 수량 제어가 필요한 도메인에서는 단순히 락을 거는 것뿐만 아니라, 데이터베이스의 **격리 수준(Isolation Level)과 스냅샷 읽기 동작 방식**을 반드시 함께 고려해야 함을 보여주는 사례입니다.
