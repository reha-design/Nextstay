# 📝 기술 도입 결정 (ADR) - 메시지 큐 시스템 (RabbitMQ)

## 1. 개요
* **주제:** 비동기 알림 처리 및 시스템 결합도(Coupling)를 낮추기 위한 메시지 큐 도입
* **결정 사항:** `RabbitMQ(AMQP 프로토콜)` 채택

---

## 2. 도입 배경 (문제 상황)
넥스트스테이(Nextstay)의 핵심 기능인 '숙소 예약 시스템'은 프론트엔드 고객에게 빠르고 안정적인 트랜잭션 응답 속도를 보장하는 것이 최우선입니다. 

하지만 예약이 성공적으로 확정된 후 이어지는 부가적인 로직들(**카카오톡 알림 전송**, **이메일 발송** 등)은 외부 API(SMTP나 알림톡 서버 등)를 호출해야 하므로 메인 서버(스프링 부트)의 응답 속도를 늦추거나, 외부 서버 장애 시 우리 메인 예약 서버까지 함께 터져버리는 **연쇄 장애(Cascading Failure)** 위험성을 내포하고 있습니다.

이를 해결하기 위해 핵심 비즈니스 로직(결제 처리 및 예약 데이터 저장)과 부가 로직(고객 알림 전송)을 물리적으로 분리하는 **비동기 메시지 큐 아키텍처**를 도입해야 했습니다.

---

## 3. 대안 비교 (RabbitMQ vs MQTT)
메시징 시스템으로 널리 쓰이는 두 가지 기술을 비교 검토했습니다.

| 비교 항목 | 🐰 RabbitMQ (AMQP 프로토콜) | 📡 MQTT 프로토콜 |
| :--- | :--- | :--- |
| **주사용처 및 목적** | 엔터프라이즈급 고신뢰성 서버 대 서버(B2B) 비동기 통신 | 저전력/불안정한 네트워크 기반의 IoT 기기 (센서 통신 등) |
| **신뢰도 및 영속성 보장** | 메시지를 디스크에 안전하게 저장하여 전송 성공을 깐깐하게 보장함 | 배터리 절약을 위해 네트워크 끊김에 관대하며 가볍게만 보냄 |
| **메시지 페이로드 크기** | 크기가 무겁고 복잡한 구조의 JSON 데이터 덩어리 전송에 적합 | 아주 작은 단위의 센서 온도 데이터 전송 등에 특화 |

---

## 4. 최종 채택 이유
> **"우리의 숙소 예약 시스템은 '안전성'과 '데이터 유실 방지'가 최우선 목표입니다."**

예약 완료 알림 등 필수 메시지가 고객에게 도달하지 않으면 치명적인 CS(고객 불만) 및 신뢰도 하락 이슈로 이어지기 때문입니다. 센서 배터리를 아낄 필요가 없으므로 페이로드 제한이 심하고 가벼운 **MQTT 대신**, 디스크 상의 메시지 영속성(Persistence) 능력이 탁월하고 엔터프라이즈 환경 및 Spring 생태계에서 완벽하게 검증된 **RabbitMQ(AMQP 프로토콜) 구조를 최종 채택**했습니다.

또한, 스프링 부트 애플리케이션 내부에 Queue 리스트로 기능만 구현하지 않고 **별도의 독립된 RabbitMQ 프로세스(서버)로 분리 구축**하여, 메인 컨트롤 타워(Spring Boot)가 다운되거나 재시작하더라도 대기 중이던 알림 데이터가 메모리에서 증발하지 않고 큐에 안전하게 보존되도록 아키텍처를 독립적으로 분리 설계했습니다.
