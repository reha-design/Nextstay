# Architecture Decision Record: 대용량 트래픽 대응을 위한 워커(Worker) Auto Scaling 전략

## 1. 배경 (Context)

미스터멘션(Nextstay)의 백그라운드 워커(Worker)는 카카오 알림톡 전송, 쿠폰 지연 발급 로직, 결제 확인 등 외부 API 연동 및 DB I/O 바운드 작업을 대규모로 비동기 처리하는 핵심 시스템입니다. 
평상시에는 트래픽이 일정하지만, **'전 회원(수백만 명) 대상 10% 쿠폰 이벤트'**나 **'성수기 프로모션 아침 푸시 알림'** 등 이벤트 프로모션이 실행되는 특정 시점에는 메시지 브로커(RabbitMQ) 큐에 수십만 건의 메시지가 순식간에 적재(Spike)되는 파괴적인 병목 현상이 발생합니다.
이 극한의 I/O 상황에서 싱글 스레드 기반 언어(예: Node.js, Bun)의 인스턴스를 운영한다면, 아무리 오토 스케일링을 하더라도 단일 프로세스 내의 이벤트 루프 한계로 인해 큐의 체공량(Queue Depth)을 즉각적으로 해소하기 어렵고 발송 지연(알림 지연) 장애가 발생할 확률이 높습니다.

따라서 워커의 개발 언어로 언어 레벨에서 제공하는 가벼운 동시성 기능인 **고루틴(Goroutine)**을 통해 압도적인 I/O 처리량을 자랑하는 **Go (Golang)**를 채택하여, 각 워커 인스턴스의 트래픽 소화 한계치를 극대화하고자 합니다.

## 2. 결정 (Decision)

워커 확장에 대한 책임을 애플리케이션 코드가 아닌 **클라우드 인프라(AWS 컨테이너 오케스트레이션)에 위임**하며, 전통적인 CPU 사용률 기반 스케일링이 아닌 **'RabbitMQ 메시지 큐 대기열 길이(Queue Depth)'를 지표(Metric)로 삼는 이벤트 기반 Auto Scaling (Event-driven Auto Scaling)** 구조를 채택합니다.

## 3. 도입 로직 및 아키텍처 (Architecture)

### 3.1. 컨테이너 기반 워커 배포 (AWS ECS / Fargate)
- Go 기반 워커 애플리케이션을 무상태(Stateless) 도커(Docker) 컨테이너로 패키징합니다.
- 가상 머신(EC2)을 부팅하는 방식은 스케일링에 수 분이 소요되므로, 수 초 만에 인스턴스를 확보할 수 있는 **Serverless 컨테이너 환경인 AWS Fargate(ECS)** 위에 배포하여 기민성을 확보합니다.

### 3.2. 이벤드 기반 스케일 아웃/인 (Metric-based Scaling)
- **트리거(Trigger)**: AWS CloudWatch 커스텀 메트릭(Custom Metric)이나 쿠버네티스 통합 환경의 경우 **KEDA**(Kubernetes Event-driven Autoscaling)를 연동합니다.
- **Rule 예시**: 
  - `Queue Messages < 1,000`: 워커 컨테이너 2대 유지 (최소 비용)
  - `Queue Messages > 10,000`: 워커 컨테이너 10대로 확장 (Scale-out)
  - `Queue Messages > 50,000`: 워커 컨테이너 30개로 파격적 확장 (Scale-out)
  - 큐가 모두 비워지면 5분의 쿨다운 타임 후 다시 워커를 2대로 축소 (Scale-in)

## 4. 결과 및 트레이드오프 (Consequences)

### 장점 (Positive)
- **비용 최적화 (Cost Efficiency)**: 평상시 잉여 리소스를 제거하여(최소 워커 가동) 스타트업/스케일업 스케일의 클라우드 인프라 요금을 획기적으로 방어합니다.
- **빠른 장애 복구 (Resilience)**: 트래픽 스파이크 시 워커 컨테이너가 수십 개로 복제되어 병렬 처리(Parallel Processing)를 수행하므로, 알림 100만 건 발송과 같은 무거운 작업도 수 분 내에 깔끔하게 소화하고 정상화됩니다.
- 싱글 스레드 런타임(Bun, Node)의 태생적 컴퓨팅 한계를 극복하는 **완벽한 수평 확장(Scale-out) 아키텍처**입니다.

### 약점 및 주의점 (Negative / Risks)
- **DB Connection 고갈 위험**: 확장된 30개의 워커가 동시에 데이터베이스(MySQL) 커넥션을 맺으려 시도할 수 있습니다. 
- **해결 방안**: 대량 확장에 대비하여 워커에서는 내부 커넥션 풀을 최소화하고, 가급적 읽기 전용 작업은 Replica DB를 바라보게 하거나 알림톡 발송 등 DB 접근 없는 로직을 워커 단일 책임으로 완벽히 분리하는 것이 중요합니다.
