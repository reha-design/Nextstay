# Architecture Decision Record: MSA 환경에서 서브 백엔드로 Bun + Elysia 도입

## 배경 (Context)

Nextstay는 호텔 예약 시스템의 다양한 측면을 처리하기 위해 마이크로서비스 아키텍처(MSA)로 설계되었습니다. 핵심 백엔드는 데이터베이스(MySQL)와의 복잡한 트랜잭션 관리, 안정성, 성숙한 생태계를 요구하기 때문에 Kotlin과 Spring Boot를 선택했습니다. 그러나 MSA 환경의 모든 서비스가 동일한 요구사항을 가지는 것은 아닙니다. 어떤 서비스는 I/O 바운드 작업 위주이거나, 빠른 확장이 필요하거나, 단순히 최소한의 보일러플레이트로 간단한 처리를 해야 할 수 있습니다.

`backend-sub` 프로젝트의 경우, 무거운 JVM 기반의 Spring Boot 애플리케이션을 보완할 수 있는 가볍고 성능이 뛰어난 대안이 필요했습니다.

## 결정 (Decision)

저희는 보조 백엔드(`backend-sub`)를 위해 **Bun** 런타임과 **Elysia.js** 프레임워크를 조합하여 사용하기로 결정했습니다.

## 도입 근거 (Rationale)

### 1. 높은 성능과 낮은 지연 시간 (High Performance & Low Latency)
Bun은 Zig 언어와 Safari의 JavaScriptCore 엔진을 사용하여 처음부터 구축된 빠르고 포괄적인 JavaScript 런타임입니다. 특히 기본적인 HTTP 처리에서 Node.js나 Deno 대비 상당한 성능 향상을 제공합니다.

**Elysia.js**는 Bun의 압도적인 속도를 활용하며, Go(Gin)나 Rust에 필적하는 성능을 지향하는 현존하는 가장 빠른 웹 프레임워크 중 하나로 벤치마크되고 있습니다. Nest.js나 Python의 FastAPI 같은 Node.js 기반 프레임워크와 비교할 때 Elysia는 훨씬 가볍고 직관적이며 오버헤드가 매우 적습니다. 서비스 간 네트워크 통신이 빈번한 MSA 환경에서, 가벼운 서비스의 요청당 처리 오버헤드를 최소화하는 것은 전체 시스템의 응답성에 매우 중요합니다.

### 2. 개발 생산성과 단순함 (Developer Productivity & Simplicity)
Bun은 툴체인의 복잡성을 크게 줄여줍니다. 런타임, 패키지 매니저(npm/yarn 대체), 번들러, 테스트 러너의 역할을 한 번에 수행합니다. 이러한 통합 시스템 덕분에 도구 설정에 들이는 시간을 줄이고 기능 개발에 더 집중할 수 있습니다. Elysia.js는 직관적인 Express 형태의 문법과 최신 TypeScript 기능을 결합하여, API 엔드포인트를 빠르게 개발할 수 있게 해줍니다. BFF(Backend-for-Frontend)나 프록시, 또는 단순한 일반 작업을 처리하는 서브 서비스에 있어 이러한 단순함은 큰 장점입니다.

### 3. 뛰어난 TypeScript 지원 및 End-to-End 타입 안정성 (Eden)
Bun과 Elysia.js는 모두 TypeScript를 염두에 두고 만들어졌습니다. Bun은 사전 컴파일 단계(`tsc`나 `ts-node` 등) 없이 TypeScript 파일을 직접 실행할 수 있어 개발 주기를 크게 단축시킵니다.

Elysia.js 생태계에서 특히 강력한 기능 중 하나가 바로 **Eden**입니다.

#### 💎 Eden: 백엔드와 프론트엔드의 '완벽한 연결'
Eden은 Elysia.js 전용으로 설계된 End-to-End 타입 세이프 클라이언트입니다. Elysia 백엔드와 프론트엔드 클라이언트(우리의 경우 Nuxt 3) 사이의 연결 고리 역할을 합니다. 데이터 패칭 로직을 수동으로 작성하거나 생성된 API 클라이언트(OpenAPI 제너레이터 등)에 의존하는 대신, Eden을 사용하면 프론트엔드에서 Elysia 서버 코드의 정확한 타입, 라우트, 데이터 전송 객체(DTO)를 직접 유추할 수 있습니다.

**Eden의 핵심 기능:**
- **Eden Treat (객체 기반 접근)**: 서버의 API 구조를 마치 로컬 함수를 호출하는 것처럼 객체 형태로 사용할 수 있게 합니다 (예: `client.api.user.get()`). 이는 완벽한 IDE 자동 완성을 지원하며, URL 오타로 인한 404 에러를 원천 차단합니다.
- **Eden Fetch**: 타입 안정성을 원하면서도 표준 `fetch` 인터페이스를 유지하고 싶을 때 사용하는 표준 fetch API의 확장입니다.
- **컴파일 타임 동기화**: Swagger나 API 명세서를 매번 확인할 필요가 없습니다. 백엔드에서 필요한 페이로드가 변경되면, 클라이언트 코드에서 즉시 타입 체크 에러로 나타납니다.

#### 이것이 Nextstay 프로젝트에서 중요한 이유
- **보일러플레이트 없는 동기화**: 백엔드(`backend-sub`)에서 응답 혹은 요청 DTO를 정의하면, 프론트엔드(`frontend`)에서 별도의 작업 없이 그 정확한 타입을 즉시 사용할 수 있습니다. 리포지토리 간에 TypeScript 인터페이스를 수동으로 복제하거나 외부 코드 매핑에 의존할 필요가 없습니다.
- **비약적인 생산성 향상**: 이러한 원활한 타입 공유는 프론트엔드와 백엔드 간의 통합 시간을 대폭 줄여줍니다. 개발자들은 프론트엔드 코드를 작성할 때 백엔드 정의로부터 직접 자동 완성, 인라인 문서화, 컴파일 타임 에러 체크 기능을 제공받습니다.
- **향상된 안정성(신뢰성)**: 백엔드의 API 라우트나 DTO를 리팩토링할 때 페이로드가 변경되면 즉시 프론트엔드에서 타입 체크 에러 플래그가 발생하므로, 마이크로서비스와 Vue 클라이언트 간의 견고한 통신을 보장합니다.

### 4. 새로운 패러다임 탐구 (Exploring New Paradigms)
Nextstay 프로젝트의 목표 중 하나는 다양한 최신 기술에 대한 실무 경험을 쌓는 것입니다. Spring Boot가 탄탄하고 전통적인 엔터프라이즈 기반을 제공하는 한편, Bun과 Elysia.js(그리고 Eden의 조합)를 통합함으로써 팀은 최신 JavaScript/TypeScript 생태계 성과를 탐구할 수 있습니다. 이러한 다양한 기술 스택 경험은 서비스 요구사항에 맞춰 특정 도구를 언제 활용해야 할지 파악하는 데 귀중한 자산이 됩니다.

## 결과 및 리스크 (Consequences)

- **장점 (Positive):**
  - JVM 서비스에 비해 시작 속도가 매우 빠르고 메모리 사용량이 극도로 적습니다.
  - 네이티브 TypeScript 지원 및 제로 설정(zero-config) 도구를 통해 개발 주기가 빠릅니다.
  - 프론트엔드(Nuxt)와 서브 백엔드 간에 타입을 매우 쉽게 공유할 수 있습니다.
- **단점 및 리스크 (Negative / Risks):**
  - Bun과 Elysia.js 생태계는 빠르게 성장하고 있지만, Spring Boot나 Node.js/Express에 비해서는 아직 새롭고 성숙도가 떨어집니다. 잘 알려지지 않은 문제에 대한 해결책이나 서드파티 라이브러리가 부족할 수 있습니다.
  - 두 번째 백엔드 언어/스택(TypeScript vs Kotlin)이 도입되므로, 개발자들에게 컨텍스트 스위칭이 요구되거나 더 넓은 범위의 기술 셋이 필요합니다.
