# 🚀 넥스트스테이(Nextstay) 서브 프로젝트 계획서

## 1. 프로젝트 개요
채용 공고의 요구사항과 우대사항을 최대한 경험해볼 수 있도록 구성한 **미니 숙박 예약 시스템**입니다. 경험이 부족한 부분을 채우고 트렌디한 기술을 어필하는 데 목적을 둡니다.

## 2. 사용 기술 스택 (Tech Stack)
* **프론트엔드**: Vue 3 (Nuxt.js), Pinia, TypeScript, **SCSS (CSS 전처리기)**
* **핵심 백엔드**: Kotlin, Spring Boot, JPA, MySQL
  * **[Java 대신 Kotlin + Spring Boot를 선택한 이유]**
  * **안정성 (NPE 해방)**: 언어 차원의 Null Safety 지원으로 서버 다운의 주범인 Null 에러 방지
  * **생산성 (코드 간결화)**: Data Class, 타입 추론 등으로 DTO 작성 등 불필요한 반복 코드(보일러플레이트) 획기적 감소
  * **완벽한 호환성**: 기존의 탄탄하고 방대한 Java / Spring 생태계 라이브러리를 제약 없이 100% 그대로 활용 가능
  * **비동기 처리 (코루틴)**: 대규모 트래픽 처리에 필수적인 비동기 로직을 동기 코드처럼 직관적이고 쉽게 작성 가능
* **서브 마이크로서비스**: **Bun + Elysia.js** (생산성/신규 기술 어필용)
* **비동기/메시지 큐**: **RabbitMQ**
* **인프라 & CI/CD**: AWS (EC2, RDS 등), GitHub Actions
* **로그 및 모니터링**: **ELK Stack** 경험 (도입)

## 3. 모르는 개념 & 궁금증 해결 (Q&A)

### Q1. ELK? 는 뭐지?
**ELK Stack**은 수많은 서버의 로그(기록)를 한곳에 효율적으로 모으고 검색, 분석하기 위한 시스템 구조입니다. (대용량 서비스 우대사항 단골 손님입니다)
* **E (Elasticsearch)**: 거대한 데이터를 저장하고 아주 빠르게 검색해주는 검색 엔진
* **L (Logstash)**: 여러 서버들로부터 로그 데이터를 수집하고 가공해서 Elasticsearch로 보내주는 시스템
* **K (Kibana)**: 저장된 로그 데이터를 모아서 예쁜 차트와 대시보드(화면)로 보여주는 도구

### Q2. 대용량 트래픽 경험은 없지만 어필할 수 있는 무언가...
실제 대규모 유저풀이 없더라도, **"대용량 트래픽을 대비한 아키텍처 구성과 부하 테스트 경험"**으로 충분히 어필할 수 있습니다.
1. **부하 테스트 (Load Testing)**: `k6`, `JMeter`, `Artillery` 같은 툴을 이용해 가상의 유저 1,000명이 동시에 접속하는 상황을 시뮬레이션하고 병목 현상(서버가 버벅이는 곳)을 파악합니다.
2. **비동기 처리 (RabbitMQ 활용)**: 예약 완료 후 DB를 업데이트하면서 동시에 이메일과 알림톡을 보내면 원래 서버는 기다리느라 뻗어버립니다. 무거운 알림 작업을 **RabbitMQ**로 빼서 비동기로 처리해 메인 서버의 부하를 줄입니다. (이게 핵심 어필 포인트입니다!)
3. **개선 과정 수치화**: 부하 테스트를 통해 "응답속도 3초 -> 0.5초로 단축, 동시 접속자 500명 -> 3,000명 수용 가능" 등으로 개선한 경험을 README에 기록합니다.

### Q3. CSS 전처리기는 뭔지 모르겠는데?
일반 CSS 작성의 단점(변수 부족, 코드 중복, 노가다)을 해결하기 위해 사용하는 **'프로그래밍 언어의 특징(변수, 함수, 곂침 등)을 가진 CSS'**입니다. 대표적으로 **SCSS(SASS), LESS**가 있습니다.
* CSS 안에서 변수를 쓰거나 `if`문, `for`문, 중첩 구조 등을 쓸 수 있게 해줍니다.
* 브라우저는 이를 바로 읽지 못하므로 컴파일러(예: Webpack, Vite)가 빌드할 때 일반 CSS로 변환해줍니다. 이번 프로젝트에서 Nuxt.js 세팅 시 **SCSS**를 함께 사용해볼 수 있습니다.

### Q4. 스프링 부트로 자바/코틀린 서버를 구성했을 때 톰캣(Tomcat) 같은 WAS가 따로 필요해?
**결론부터 말씀드리면 "필요하지 않습니다!"** 
* 과거의 전통적인 Spring(Legacy) 방식에서는 개발자가 코드를 `.war` 파일로 빌드한 다음, 서버에 설치된 **외장 Tomcat(WAS)** 폴더 안에 직접 밀어 넣고 톰캣을 실행시켜야 했습니다.
* 하지만 **Spring Boot**는 이름 그대로 "부트(실행)" 시키기 위해 **내장 톰캣(Embedded Tomcat)**을 기본적으로 포함하고 있습니다.
* 따라서 별도의 WAS 설치 없이, 그냥 자바 실행 명령어(`java -jar app.jar`) 하나만 치면 스프링 부트가 알아서 내장된 톰캣을 띄우고 서버를 실행해 줍니다. 
* 이것이 기존 Spring 프레임워크와 Spring Boot의 가장 강력하고 편리한 차이점 중 하나입니다.

#### Q4-1. 그러면 컴파일(빌드)이 필요하지 않은 거야?
**아닙니다! 컴파일과 빌드는 무조건 필요합니다.**
* **컴파일(빌드):** 우리가 짠 Kotlin/Java 코드를 컴퓨터(JVM)가 읽을 수 있는 기계어(Bytecode)로 번역하고 압축하는 과정입니다. 이 과정을 거치면 `.jar` 라는 하나의 실행 파일이 만들어집니다.
* **내장 WAS의 의미:** 이 만들어진 `.jar` 파일 안에 **"톰캣 서버 프로그램"** 자체가 통째로 같이 압축되어 들어간다는 뜻입니다. 
* 즉, 코드를 컴파일(빌드)하는 과정은 여전히 존재하지만, 그 결과물을 빈 컴퓨터에 가져가서 실행만 하면(별도로 톰캣을 설치할 필요 없이) 켜진다는 배포의 편리함을 의미합니다.

## 4. 아키텍처 및 전체 흐름 시나리오

1. **서비스 접속**: 유저가 Nuxt.js 클라이언트에 접속합니다 (TypeScript, SCSS 적용).
2. **핵심 로직 처리 (Spring Boot + Kotlin)**: 유저가 숙소 목록을 조회하고 예약을 진행합니다 (MySQL 사용).
3. **신규 기술 맛보기 (Bun + Elysia.js)**: 방문자 접속 통계 추합이나 가벼운 실시간 상태 확인 등 빠르고 가벼워도 되는 API를 별도로 분리해, 최신 기술 도입 능력을 어필합니다.
4. **비동기 메세지 통신 (RabbitMQ)**: 예약을 완료하면, Spring Boot가 RabbitMQ 큐에 "예약 완료 알림 전송해!" 라는 메세지만 남기고 바로 유저에게 성공 화면을 보여줍니다 (비동기 처리).
5. **로그/모니터링 (ELK)**: Spring Boot 및 Elysia 서버에서 발생하는 모든 에러와 API 호출 기록을 ELK Stack 인프라로 보내서 분석 대시보드를 구축합니다.
6. **배포 (GitHub Actions + AWS)**: 코드를 GitHub 메인 브랜치에 푸시하면, GitHub Actions가 자동으로 빌드 테스트를 거쳐 AWS 서버에 코드를 배포합니다 (CI/CD).
